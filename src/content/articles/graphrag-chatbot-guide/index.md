GraphRAGは、文書を小さな断片として検索するだけでなく、文書中のエンティティと関係を知識グラフとして構造化し、データ全体のまとまりも事前に要約するRAG手法です。

通常のベクトル検索型RAGが得意な「質問と意味の近い文章を探す」処理を補い、複数の文書に分散した関係をたどる質問や、文書群全体の傾向をまとめる質問に対応しやすくなります。一方で、索引作成の費用、更新の複雑さ、検索方式の選択など、導入前に考えるべき点も増えます。

この記事ではMicrosoftが公開しているGraphRAGを基準に、仕組みとチャットボットへ導入するための現実的な設計を整理します。

## 前提条件

| 項目               | 内容                                                                 |
| ------------------ | -------------------------------------------------------------------- |
| 対象               | Microsoft GraphRAG                                                   |
| 確認バージョン     | `v3.1.0`                                                             |
| 最新リリース確認日 | 2026-06-12                                                           |
| Python             | 3.10から3.12                                                         |
| 想定用途           | 社内文書、調査資料、問い合わせ履歴などを回答根拠にするチャットボット |

GraphRAGは更新が活発なプロジェクトです。公式ドキュメントは、マイナーバージョン間でも`graphrag init --root [path] --force`を実行して設定形式を更新するよう案内しています。実運用ではバージョンを固定し、更新前に設定とプロンプトをバックアップしてください。

## 通常のRAGが苦手とする質問

一般的なRAGは、文書をチャンクへ分割し、それぞれをEmbeddingへ変換してベクトルデータベースへ保存します。質問が来たら、意味の近い上位`k`件のチャンクをLLMへ渡します。

```text
質問
  ↓
Embedding
  ↓
ベクトル検索
  ↓
上位の文章チャンク
  ↓
LLMが回答
```

この方式は、回答が一つか数個の文章に明確に書かれている質問に向いています。

- 製品Aの保証期間は何年か
- 山田さんが担当するプロジェクトは何か
- 手順書に記載された再起動方法は何か

一方、次の質問では、質問文と意味の近いチャンクを数件集めるだけでは情報が不足しやすくなります。

- 顧客から寄せられた不満の主要テーマは何か
- 複数の報告書に登場する企業と人物はどう関係しているか
- プロジェクトの遅延に影響した要因を部門横断で整理してほしい
- 1年間の問い合わせ履歴から、新しく増えている問題を説明してほしい

一つ目の問題は、情報同士のつながりをたどる必要があることです。二つ目は、データセット全体を集約して理解する必要があることです。GraphRAGは、知識グラフと階層的な要約を作ることで、この二つに対応します。

## GraphRAGの索引作成

GraphRAGは、質問を受け取る前に文書を解析し、検索用の索引を作ります。Microsoft GraphRAGの標準パイプラインは、概ね次の処理を行います。

```text
入力文書
  ↓
TextUnitへ分割
  ↓
エンティティ、関係、主張を抽出
  ↓
知識グラフを構築
  ↓
Leiden法でコミュニティを階層化
  ↓
コミュニティごとの要約レポートを生成
  ↓
テキストとエンティティのEmbeddingを生成
  ↓
Parquetテーブルとベクトルストアへ保存
```

### TextUnit

入力文書を分析可能な単位へ分割したものです。通常のRAGにおけるチャンクに近い役割を持ち、回答の根拠となる原文へ戻るためにも使われます。

### エンティティと関係

LLMがTextUnitから人物、組織、場所、製品、概念などのエンティティを抽出します。さらに「人物Aが組織Bに所属する」「障害Cが機能Dへ影響した」といった関係や、重要な主張を抽出します。

抽出結果は完全な事実とは限りません。LLMが名称を誤って統合したり、同じ対象を別エンティティとして扱ったりする可能性があるため、重要用途では品質確認が必要です。

### コミュニティ

関係の密なエンティティ群を、Leiden法によってコミュニティへまとめます。さらにコミュニティを階層化することで、細かな話題からデータセット全体の大きなテーマまで、異なる粒度で扱えるようにします。

### コミュニティレポート

各コミュニティについて、主要なエンティティ、関係、主張をLLMで要約します。この事前生成された要約が、データ全体について尋ねるGlobal Searchの重要な材料になります。

索引作成ではLLMを何度も呼び出すため、通常のベクトルRAGより時間と費用が大きくなる傾向があります。公式ドキュメントも、小さなチュートリアルデータと低価格なモデルから試すよう注意しています。

## 4つの検索方式

Microsoft GraphRAGには、用途の異なる検索方式があります。チャットボットでは、すべての質問を一つの方式へ送るのではなく、質問の性質に応じて選択するのが重要です。

| 方式          | 適した質問                                       | 主に使う情報                               | 特徴                                |
| ------------- | ------------------------------------------------ | ------------------------------------------ | ----------------------------------- |
| Basic Search  | 原文に答えが直接書かれている質問                 | 上位`k`件のTextUnit                        | 通常のベクトルRAGに近く、比較的軽い |
| Local Search  | 特定の人物、製品、組織、出来事に関する質問       | エンティティ、関係、原文、コミュニティ情報 | 対象を起点に周辺情報を広げる        |
| Global Search | 文書群全体のテーマ、傾向、要因を問う質問         | コミュニティレポート                       | Map-Reduce形式で全体像をまとめる    |
| DRIFT Search  | 特定の話題から始めつつ、関連領域も広く調べる質問 | Local Searchとコミュニティ情報             | 詳細さと広がりのバランスを取る      |

### Local Search

質問と意味的に関連するエンティティを入口として、接続されたエンティティ、関係、コミュニティレポート、対応する原文チャンクを集めます。候補を優先順位付けし、コンテキストウィンドウへ収まるように絞ってから回答を生成します。

例えば「製品Alphaで発生した障害と、影響を受けた顧客を説明して」のような、対象が具体的な質問に向いています。

### Global Search

指定した階層のコミュニティレポートを複数のバッチへ分け、それぞれから中間回答と重要度を生成します。重要な中間回答を集約し、最終回答を作るMap-Reduce方式です。

「問い合わせ全体に共通する問題を5つ挙げて」のような質問に向いています。ただし、複数回のLLM呼び出しが必要となり、検索費用と応答時間が大きくなりやすい方式です。

### DRIFT Search

DRIFTは **Dynamic Reasoning and Inference with Flexible Traversal** の略です。コミュニティの広い視点を使って検索の開始地点を改善し、そこから詳細な追加質問とローカル探索を進めます。

対象が具体的でも、質問に答えるには周辺の話題を広く調べる必要がある場合に候補となります。

## チャットボットへ導入する構成

実用的なチャットボットでは、GraphRAGをユーザー画面から直接呼び出すのではなく、アプリケーションサーバーの検索層として配置します。

```text
ユーザー
  ↓
チャットUI
  ↓
認証・認可
  ↓
質問分類 / 検索ルーター
  ├─ Basic Search
  ├─ Local Search
  ├─ Global Search
  └─ DRIFT Search
       ↓
回答生成
  ↓
引用・権限・安全性の検査
  ↓
ストリーミング応答
```

索引作成はオンラインの質問処理から分離します。

```text
文書ストレージ
  ↓
前処理・アクセス権付与
  ↓
GraphRAG Indexer
  ↓
Parquet / ベクトルストア
  ↓
バージョン付き索引
```

この分離により、重い索引作成がチャットの応答を止めることを防ぎます。新しい索引を完成させて評価した後、チャットボットが参照する索引バージョンを切り替えます。

## 段階的な導入手順

### 1. GraphRAGが必要な質問を定義する

最初に、通常のRAGで失敗している質問を収集します。GraphRAGを使う理由を「精度が上がりそうだから」だけにせず、次のように分類します。

- 複数文書をまたぐ関係の探索
- 文書群全体のテーマや傾向の集約
- 特定エンティティから関連情報を広げる探索
- 長い調査対話での追加質問

回答が一つの文書に直接書かれているFAQが中心なら、通常のRAGの改善を先に行う方が安価で単純です。

### 2. 小さな検証用データを用意する

本番の全文書を最初から投入せず、代表的な文書と評価質問を選びます。

- 50から200件程度の代表文書
- Basic、Local、Globalに分けた質問
- 期待する答えと根拠文書
- 回答してはいけない権限外の質問
- 情報不足として回答を控えるべき質問

この評価セットは、GraphRAG導入後もモデルや設定を変更するたびに使います。

### 3. CLIで索引と検索を試す

Python 3.10から3.12の仮想環境を作り、GraphRAGをインストールします。本番では記事執筆時点の確認バージョンへ固定することを推奨します。

```powershell title="Windowsでの検証環境"
mkdir graphrag-poc
cd graphrag-poc
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install graphrag==3.1.0
graphrag init
```

`graphrag init`は、主に次のファイルとディレクトリを生成します。

```text
graphrag-poc/
├── .env
├── settings.yaml
└── input/
```

`.env`の`GRAPHRAG_API_KEY`へ利用するLLMサービスのAPIキーを設定し、検証文書を`input`へ配置します。APIキーはGitへコミットしないでください。

```powershell title="索引作成"
graphrag index
```

索引作成後、検索方式を変えて同じ質問を比較します。

```powershell title="全体傾向を尋ねる"
graphrag query "問い合わせ全体で繰り返し報告されている問題は何ですか"
```

```powershell title="特定対象について尋ねる"
graphrag query "製品Alphaの障害と関係者を説明してください" --method local
```

CLIは動作確認に便利ですが、チャットボット本体から毎回別プロセスとして起動する構成は、応答時間、同時実行、エラー処理、監視の面で不利です。本番ではPython APIを使う検索サービスとして組み込む方が管理しやすくなります。

### 4. 質問を振り分ける

単純なルールから始め、必要に応じて小さな分類モデルやLLMルーターへ発展させます。

```python title="検索方式を選ぶ概念例"
def choose_search_method(question: str) -> str:
    if asks_about_dataset_wide_themes(question):
        return "global"
    if names_a_specific_entity(question):
        return "local"
    if needs_broad_exploration_from_a_specific_topic(question):
        return "drift"
    return "basic"
```

ルーターが迷った場合は、安価なBasic SearchまたはLocal Searchから始め、得られた根拠が不足しているときだけDRIFTやGlobal Searchへ広げる方法もあります。

<aside>
  <strong>実務上の推奨:</strong> Global Searchをすべての質問の既定値にすると、費用と待ち時間が増えます。全体集約が必要な質問へ限定してください。
</aside>

### 5. 会話履歴を検索用に整える

「それに関係した別の障害は？」のような追加質問は、そのままでは対象が分かりません。会話履歴から独立した検索質問へ書き換えます。

```text
直前の質問:
製品Alphaの認証障害について教えて

追加質問:
それに関係した別の障害は？

検索用に書き換え:
製品Alphaの認証障害と関係する別の障害は何か？
```

履歴全体を無制限に検索へ渡すのではなく、現在の質問に必要なエンティティ、期間、条件だけを抽出します。GraphRAGのLocal SearchとGlobal Searchは会話履歴を扱えますが、アプリケーション側でも履歴の長さと個人情報を管理する必要があります。

### 6. 引用を回答の一部にする

知識グラフは検索のための派生データです。最終的な根拠として、元のTextUnitや文書への参照を利用者へ示します。

回答APIは、文章だけでなく次のような構造を返すと扱いやすくなります。

```json title="回答APIの例"
{
  "answer": "製品Alphaの障害は認証基盤の変更と関連しています。",
  "searchMethod": "local",
  "sources": [
    {
      "documentId": "incident-2026-041",
      "title": "認証障害報告書",
      "excerpt": "変更適用後にトークン検証エラーが増加した",
      "url": "/documents/incident-2026-041"
    }
  ],
  "indexVersion": "2026-06-12T03:00:00Z"
}
```

引用先にも利用者のアクセス権を適用します。検索できたからといって、利用者へ表示してよいとは限りません。

### 7. 索引更新を運用する

文書更新のたびに大規模な索引を作り直すと、費用と反映時間が問題になります。文書の性質に応じて更新方式を決めます。

| データの性質                     | 更新方法の例                                                  |
| -------------------------------- | ------------------------------------------------------------- |
| 月次レポートなど更新が少ない     | 夜間や週次の一括再索引                                        |
| 問い合わせ履歴など継続的に増える | 差分更新と定期的な全体再構築                                  |
| 規程など正確性が重要             | 承認済み文書だけを索引化し、公開前に評価                      |
| 削除要求がある個人データ         | 文書、TextUnit、グラフ、Embeddingから削除できる追跡情報を保持 |

索引にはバージョンを付け、チャットのログへ利用した索引バージョンを記録します。問題が起きたときに、どのデータと設定で回答したかを追跡できます。

## 検索ルーターを含むAPI設計

チャットボットのバックエンドは、次の責務に分けると変更しやすくなります。

```text
Chat API
├── Conversation Service
├── Query Rewriter
├── Search Router
├── GraphRAG Search Service
├── Citation Builder
├── Authorization Filter
└── Evaluation / Logging
```

GraphRAG固有のAPIをChat API全体へ広げず、Search Serviceの内側へ閉じ込めます。こうすると、質問によって通常のベクトルRAGやデータベース検索を併用できます。

```python title="チャット処理の概念例"
async def answer_chat(user, conversation, message):
    query = await rewrite_as_standalone_question(conversation, message)
    method = await route_query(query)

    context = await graphrag_search(
        query=query,
        method=method,
        allowed_document_ids=user.allowed_document_ids,
    )

    answer = await generate_grounded_answer(query, context)
    return attach_citations(answer, context)
```

これは設計を示す疑似コードです。実際のGraphRAG Python APIはバージョンによって変わる可能性があるため、導入時に固定したバージョンの公式APIとサンプルNotebookを確認してください。

## 評価すべき項目

GraphRAGの導入効果は、回答が自然に見えるかだけでは判断できません。通常のRAGを基準として、同じ質問セットで比較します。

| 観点     | 確認内容                                   |
| -------- | ------------------------------------------ |
| 正確性   | 期待した事実を含むか                       |
| 根拠性   | 回答が引用元で裏付けられているか           |
| 網羅性   | 全体質問で重要なテーマを漏らしていないか   |
| 関係探索 | 複数文書に分散した関係を正しく結べるか     |
| 拒否     | 根拠がない場合に推測せず回答を控えられるか |
| 権限     | 閲覧権限外の情報を回答や引用へ含めないか   |
| 応答時間 | 検索方式ごとの待ち時間が許容範囲か         |
| 費用     | 索引作成と質問1件あたりのLLM利用量         |

Global Searchでは、利用するコミュニティ階層によって詳細さ、トークン量、応答時間が変わります。細かい階層を広く読むほど網羅的になりやすい一方、費用も増えます。

本番導入前には、少なくとも次の比較を行います。

1. LLMだけで回答
2. Basic Search
3. Local Search
4. Global Search
5. 必要ならDRIFT Search

すべての質問でGraphRAGが勝つ必要はありません。質問ごとに、最も単純で十分な方式を選べることが重要です。

## セキュリティと運用上の注意

### グラフにもアクセス権が必要

元文書へのアクセス権だけでなく、文書から抽出されたエンティティ、関係、コミュニティレポートにも情報が残ります。異なる権限の文書を同じグラフへ統合すると、関係の存在だけが漏れる可能性があります。

権限境界が厳しい場合は、テナント、部署、機密区分ごとに索引を分ける方が安全です。共有グラフを使う場合は、検索結果と引用だけでなく、派生したグラフ情報にも認可フィルターを適用します。

### プロンプトインジェクションを想定する

入力文書内に「以前の指示を無視せよ」といった文章が含まれる可能性があります。取得文書を命令ではなくデータとして扱うようシステムプロンプトで明示し、外部操作を行うエージェントとは権限を分離します。

### 抽出結果を真実とみなさない

知識グラフはLLMが生成した索引です。原文にない関係を生成したり、同名の人物を統合したりする可能性があります。重要な判断では原文を確認し、回答へ引用を付けます。

### 費用を二つに分けて管理する

GraphRAGでは、次の費用を別々に計測します。

- 文書の追加や更新時に発生する索引作成費用
- 利用者の質問ごとに発生する検索・回答費用

Global SearchはMap-Reduce形式で複数回LLMを呼び出すため、質問単価が高くなる可能性があります。キャッシュ、質問ルーター、同時実行数、トークン上限を設定します。

## どのような場合に採用するか

GraphRAGは、次の条件が多く当てはまる場合に検討する価値があります。

- 複数文書をまたぐ人物、組織、出来事の関係が重要
- 文書群全体の傾向やテーマを質問したい
- 通常のRAGで関連チャンクは取れるが、情報をつなげられない
- 索引作成に時間とLLM費用をかけられる
- 評価質問と正解例を用意できる
- 文書更新とアクセス権を管理できる

次の場合は、まず通常のRAGを改善する方が適しています。

- FAQや手順書など、一つの文章に答えがある
- 文書数が少なく、全文をコンテキストへ入れられる
- 文書が頻繁に変わり、すぐに反映する必要がある
- 回答までの遅延や費用を最小化したい
- エンティティ間の関係や全体要約を必要としない

## まとめ

GraphRAGの中心は、単にグラフデータベースを検索することではありません。LLMで文書から知識グラフを作り、コミュニティを階層化し、その要約を検索時に利用することで、局所的な関係探索とデータ全体の理解を補います。

チャットボットへ導入するときは、次の順番が現実的です。

1. 通常のRAGが失敗する質問を特定する
2. 小さなデータと評価セットでGraphRAGを検証する
3. Basic、Local、Global、DRIFTを質問ごとに使い分ける
4. GraphRAGを独立した検索サービスとして組み込む
5. 原文への引用、認可、索引バージョンを回答へ付ける
6. 品質、応答時間、索引費用、質問費用を継続評価する

GraphRAGを通常のRAGの完全な置き換えと考えるより、複雑な関係探索と全体集約を担当する検索手段として追加する方が、チャットボットの費用と品質を制御しやすくなります。
