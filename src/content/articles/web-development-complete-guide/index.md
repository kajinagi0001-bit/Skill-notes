Webサイト構築は、画面を作る作業だけではありません。目的と利用者を定義し、適切な構成を選び、安全で使いやすい実装を行い、テストして公開し、公開後も監視・更新する一連の活動です。

この記事では、初めてWebサイトを構築する人が全体像をつかみ、必要な技術を順序立てて学べるように整理します。特定のフレームワークの操作手順ではなく、技術が変わっても使える設計判断を中心に扱います。

| 項目                 | 内容                                    |
| -------------------- | --------------------------------------- |
| 対象                 | 一般的なWebサイト・Webアプリケーション  |

## 最初に決めること

コードを書く前に、サイトの目的と制約を明文化します。ここが曖昧なまま技術を選ぶと、不要な機能や運用負担が増えます。

### 目的

代表的な目的は次のとおりです。

- 会社、製品、店舗などの情報を公開する
- 技術記事やニュースを継続的に掲載する
- 問い合わせや資料請求を受け付ける
- 商品やサービスを販売する
- 会員向けの機能を提供する
- 社内業務をWeb上で処理する
- データを検索・分析・可視化する

目的は「Webサイトを作る」ではなく、「誰が何を達成できるようにするか」で表現します。

```text
悪い例:
会社のWebサイトを新しくする

改善例:
採用候補者が職種、働き方、選考手順を理解し、
スマートフォンから応募できるようにする
```

### 利用者

次の条件を整理します。

- 主な利用者と利用目的
- PC・スマートフォンの利用比率
- 対応する言語や地域
- 通信速度や端末性能
- アカウント登録の有無
- 障害のある利用者への対応
- 社内限定か、インターネット公開か

### 必要なページと機能

機能を優先度別に分けます。

| 優先度 | 意味                 |
| ------ | -------------------- |
| Must   | 初回公開に必須       |
| Should | 重要だが後日追加可能 |
| Could  | 余裕があれば追加     |
| Won't  | 今回は実装しない     |

初回公開ではMustに絞り、実際の利用状況を確認してから拡張する方が、推測だけで大規模に作るより失敗を抑えられます。

### 非機能要件

画面に直接現れない要件も早い段階で決めます。

- 想定アクセス数
- 表示速度
- 稼働率
- セキュリティ
- バックアップと復旧
- アクセシビリティ
- 対応ブラウザ
- 法令・プライバシー対応
- 運用担当者と更新頻度
- 開発・運用予算

## Webサイトが表示される仕組み

利用者がURLを開くと、概ね次の処理が行われます。

```text
1. ブラウザへURLを入力
2. DNSでドメイン名から接続先を調べる
3. HTTPSでサーバーへ接続する
4. HTTPリクエストを送信する
5. サーバーがHTMLなどを返す
6. ブラウザがHTMLを解析する
7. CSS、JavaScript、画像などを追加取得する
8. 画面を描画してJavaScriptを実行する
```

主な用語は次のとおりです。

| 用語        | 役割                                     |
| ----------- | ---------------------------------------- |
| URL         | Web上の資源の場所を表す                  |
| ドメイン    | 人が扱いやすいサイト名                   |
| DNS         | ドメインと接続先を対応付ける             |
| HTTP        | ブラウザとサーバーがデータを交換する規則 |
| HTTPS       | TLSで保護されたHTTP通信                  |
| Webサーバー | HTTPリクエストへ応答する                 |
| ブラウザ    | HTML、CSS、JavaScriptを解釈して表示する  |
| CDN         | 複数地域から静的ファイルを高速配信する   |

## Web構築の基本技術

ブラウザ側の基本はHTML、CSS、JavaScriptです。

```text
HTML       構造と意味
CSS        見た目とレイアウト
JavaScript 動作とデータ処理
```

役割を混同せず、HTMLだけでも内容と操作が理解できる状態を基礎にすると、アクセシビリティや保守性を高めやすくなります。

## HTMLで構造を作る

HTMLはページの内容と意味を表します。見た目だけを目的に要素を選ばず、内容に合う要素を使います。

```html title="index.html"
<!doctype html>
<html lang="ja">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <meta
      name="description"
      content="Web構築の基本を紹介するサンプルサイトです。"
    />
    <title>Web構築ガイド</title>
    <link rel="stylesheet" href="./styles.css" />
    <script src="./main.js" type="module"></script>
  </head>
  <body>
    <header>
      <a href="/">Web構築ガイド</a>
      <nav aria-label="メインナビゲーション">
        <a href="#about">概要</a>
        <a href="#contact">問い合わせ</a>
      </nav>
    </header>

    <main>
      <section id="about">
        <h1>Webサイト構築を始める</h1>
        <p>HTML、CSS、JavaScriptの役割から学びます。</p>
        <button id="show-details" type="button">詳細を表示</button>
        <p id="details" hidden>小さなサイトから段階的に構築します。</p>
      </section>
    </main>

    <footer>
      <small>&copy; 2026 Web構築ガイド</small>
    </footer>
  </body>
</html>
```

### セマンティックHTML

内容の役割を表す要素を使います。

- `<header>`: ページやセクションの導入部
- `<nav>`: 主要なナビゲーション
- `<main>`: ページの主要内容
- `<article>`: 単独で配布可能な記事
- `<section>`: 見出しを持つ意味上のまとまり
- `<aside>`: 補足情報
- `<footer>`: 末尾情報
- `<button>`: 操作を実行するボタン
- `<a>`: 別の場所へ移動するリンク

クリック処理が必要だからという理由で`<div>`をボタン代わりにすると、キーボード操作や支援技術への対応を自分で再実装する必要があります。標準HTML要素を優先します。

### 見出し

見出しはデザイン上の大きさではなく、文書構造に沿って使います。

```text
h1 ページの主題
├─ h2 大きな章
│  ├─ h3 小さな節
│  └─ h3 小さな節
└─ h2 大きな章
```

通常、ページの主題となる`<h1>`を一つ用意し、見出しレベルを不自然に飛ばさないようにします。

### 画像

意味のある画像には、内容や目的を伝える`alt`を設定します。

```html
<img
  src="./images/dashboard.webp"
  alt="月別売上を棒グラフで表示した管理画面"
  width="1200"
  height="675"
  loading="lazy"
/>
```

装飾だけの画像は`alt=""`とし、支援技術に読み上げさせないようにします。`width`と`height`を指定すると、読み込み前に表示領域を確保でき、レイアウトのずれを抑えられます。

### フォーム

入力欄にはラベルを関連付けます。

```html
<form method="post" action="/contact">
  <div>
    <label for="email">メールアドレス</label>
    <input id="email" name="email" type="email" autocomplete="email" required />
  </div>

  <div>
    <label for="message">問い合わせ内容</label>
    <textarea id="message" name="message" required></textarea>
  </div>

  <button type="submit">送信する</button>
</form>
```

ブラウザの入力検証は利用者を支援しますが、セキュリティ対策にはなりません。送信された値は、サーバー側でも必ず検証します。

## CSSで見た目とレイアウトを作る

CSSは、色、余白、文字、レイアウト、画面幅への適応を担当します。

```css title="styles.css"
:root {
  color-scheme: light dark;
  --background: #ffffff;
  --surface: #f4f6f8;
  --text: #172033;
  --accent: #2457d6;
  --content-width: 70rem;
}

* {
  box-sizing: border-box;
}

body {
  margin: 0;
  background: var(--background);
  color: var(--text);
  font-family: system-ui, sans-serif;
  line-height: 1.7;
}

header,
main,
footer {
  width: min(calc(100% - 2rem), var(--content-width));
  margin-inline: auto;
}

header {
  display: flex;
  padding-block: 1rem;
  align-items: center;
  justify-content: space-between;
  gap: 1rem;
}

nav {
  display: flex;
  gap: 1rem;
}

button {
  min-height: 2.75rem;
  padding-inline: 1rem;
  border: 0;
  border-radius: 0.4rem;
  background: var(--accent);
  color: white;
  cursor: pointer;
}

:focus-visible {
  outline: 3px solid var(--accent);
  outline-offset: 3px;
}

@media (max-width: 40rem) {
  header {
    align-items: flex-start;
    flex-direction: column;
  }
}
```

### ボックスモデル

要素の大きさは、概念的に次の領域から構成されます。

```text
margin
  border
    padding
      content
```

`box-sizing: border-box`を指定すると、設定した幅と高さに`padding`と`border`が含まれるため、レイアウトを計算しやすくなります。

### レイアウト

主に次を使い分けます。

| 技術       | 適した用途                               |
| ---------- | ---------------------------------------- |
| 通常フロー | 文書を上から順に配置                     |
| Flexbox    | 一方向の配置、ナビゲーション、整列       |
| Grid       | 行と列を持つ二次元レイアウト             |
| Position   | 重ね合わせ、固定表示、基準位置からの配置 |

最初からすべてを絶対配置すると、文章量や画面幅の変化に対応できません。通常フロー、Flexbox、Gridを基礎にします。

### レスポンシブデザイン

特定端末の幅を固定的に再現するのではなく、利用可能な領域へ自然に適応させます。

- 固定幅ではなく`max-width`を使う
- 画像に`max-width: 100%`を指定する
- 小さい画面を基準に設計する
- コンテンツが崩れる位置でメディアクエリを追加する
- 文字を極端に小さくしない
- 横長の表やコードは横スクロールを許可する
- タップ対象へ十分な大きさと間隔を設ける

### CSSの管理

規模が大きくなるほど、命名と責務分離が重要になります。

- グローバルな基本スタイル
- レイアウト
- 再利用可能なコンポーネント
- ページ固有のスタイル
- デザイントークン

値を無秩序に直接記述せず、色、余白、文字サイズなどをCSSカスタムプロパティとして管理すると、変更しやすくなります。

## JavaScriptで動作を追加する

JavaScriptは、利用者の操作への応答、サーバーとの通信、画面の部分更新などを担当します。

```javascript title="main.js"
const button = document.querySelector("#show-details");
const details = document.querySelector("#details");

button?.addEventListener("click", () => {
  const willShow = details.hidden;
  details.hidden = !willShow;
  button.textContent = willShow ? "詳細を隠す" : "詳細を表示";
});
```

### JavaScriptへ依存しすぎない

記事本文、連絡先、主要ナビゲーションなど、静的に提供できる情報はHTMLとして返します。JavaScriptは必要な機能へ限定します。

JavaScriptの読み込み失敗や実行遅延が起きても、最低限の内容を読める設計を**プログレッシブエンハンスメント**と呼びます。

### DOM操作

DOMは、ブラウザがHTMLをオブジェクトとして表したものです。JavaScriptはDOMを通して要素を検索・変更します。

```javascript
const message = document.querySelector("#message");
message.textContent = "更新しました";
```

信頼できない文字列をHTMLとして挿入すると、クロスサイトスクリプティング（XSS）の原因になります。単純な文字表示には`textContent`を使い、HTMLを許可する場合は十分に検証・無害化します。

### 非同期通信

`fetch()`でAPIへHTTPリクエストを送れます。

```javascript
async function loadArticles() {
  const response = await fetch("/api/articles");

  if (!response.ok) {
    throw new Error(`Request failed: ${response.status}`);
  }

  return response.json();
}
```

実装時には次を考慮します。

- 読み込み中の表示
- タイムアウト
- 通信失敗
- 再試行
- キャンセル
- 空データ
- 認証切れ
- 二重送信

### 状態管理

画面が複雑になると、現在の利用者、入力値、取得データ、表示状態などの管理が必要になります。状態の保存場所を明確にします。

| 状態                       | 保存場所の例                     |
| -------------------------- | -------------------------------- |
| 一時的な開閉状態           | コンポーネント内                 |
| URLで共有すべき検索条件    | クエリ文字列                     |
| ログイン状態               | サーバーセッション、安全なCookie |
| サーバーから取得したデータ | APIキャッシュ                    |
| 永続的な業務データ         | データベース                     |

すべてをグローバル変数や`localStorage`へ保存すると、同期、期限、セキュリティの問題が生じます。

## Webサイトの構成を選ぶ

サイトの目的に応じて、生成・配信方法を選びます。

### 静的サイト

事前に生成したHTML、CSS、JavaScriptを配信します。

適した用途:

- 会社案内
- 技術ブログ
- ドキュメント
- ポートフォリオ
- キャンペーンページ

長所:

- 高速に配信しやすい
- サーバー処理が少なく、攻撃面が小さい
- 運用費用を抑えやすい
- CDNと相性がよい

制約:

- 利用者ごとに異なる画面
- リアルタイム更新
- ログイン後の複雑な処理

このSkill NotesはAstroで静的HTMLを生成し、GitHub Pagesへ配信する構成です。

### サーバーサイドレンダリング

リクエスト時にサーバーがHTMLを生成します。

適した用途:

- ログイン利用者ごとに表示が変わる
- データが頻繁に更新される
- リクエスト時の権限判定が必要
- 検索エンジンへHTMLを確実に提供したい

サーバー、キャッシュ、障害対応、スケーリングなどの運用が必要になります。

### SPA

最初にJavaScriptアプリケーションを読み込み、その後はブラウザ上で画面を切り替えます。

適した用途:

- 操作の多い管理画面
- リアルタイムなUI
- デスクトップアプリに近い操作性

JavaScript量、初期表示、アクセシビリティ、URL管理、SEOへの配慮が必要です。一般的な情報サイトを無条件にSPAにする必要はありません。

### ハイブリッド構成

ページごとに静的生成、サーバーレンダリング、クライアント処理を使い分けます。現在のWebフレームワークには、この構成を支援するものがあります。

「最新だから」ではなく、ページごとのデータ更新頻度と対話性で判断します。

## フレームワークを使うか

HTML、CSS、JavaScriptだけでもサイトは構築できます。フレームワークは、複雑さを自動的に消すものではなく、特定の複雑さを管理するための仕組みです。

フレームワークが有効になりやすい条件:

- 多数の画面でUIを再利用する
- 状態変化が多い
- ルーティングやデータ取得が複雑
- 複数人で規約を共有する
- 静的生成とサーバー処理を統合したい

不要な可能性が高い条件:

- 数ページの情報サイト
- JavaScriptをほとんど必要としない
- 更新頻度が低い
- ビルド環境を維持したくない

技術選定では、次を比較します。

- 要件を満たせるか
- チームが保守できるか
- 安定した更新方針があるか
- セキュリティ更新を追えるか
- テストとデプロイを自動化できるか
- 移行コストを許容できるか

## バックエンドの役割

バックエンドはブラウザから直接実行させるべきでない処理を担当します。

- 認証と認可
- データベース操作
- 入力検証
- メール送信
- 決済処理
- ファイル処理
- 外部APIとの通信
- 秘密情報を使う処理
- 監査ログ

```text
ブラウザ
  ↓ HTTPS
Web/APIサーバー
  ├─ 認証・認可
  ├─ 業務ロジック
  ├─ データベース
  └─ 外部サービス
```

APIキーやデータベースのパスワードをブラウザへ配信するJavaScriptに埋め込んではいけません。ブラウザへ送った情報は利用者が確認できます。

## APIを設計する

Web APIでは、URL、HTTPメソッド、ステータスコード、入出力形式を定義します。

```text
GET    /api/articles        記事一覧を取得
GET    /api/articles/42     記事を取得
POST   /api/articles        記事を作成
PATCH  /api/articles/42     記事の一部を更新
DELETE /api/articles/42     記事を削除
```

成功時に常に`200`を返すのではなく、処理結果に合うステータスコードを使います。

| コード                      | 用途の例             |
| --------------------------- | -------------------- |
| `200 OK`                    | 取得・更新成功       |
| `201 Created`               | 作成成功             |
| `204 No Content`            | 本文を返さない成功   |
| `400 Bad Request`           | リクエスト形式が不正 |
| `401 Unauthorized`          | 認証が必要           |
| `403 Forbidden`             | 権限不足             |
| `404 Not Found`             | 対象がない           |
| `409 Conflict`              | 現在の状態と競合     |
| `422 Unprocessable Content` | 内容の検証に失敗     |
| `429 Too Many Requests`     | リクエスト過多       |
| `500 Internal Server Error` | サーバー内部の失敗   |

APIには次も必要です。

- 入出力スキーマ
- エラー形式
- ページネーション
- 並べ替えと絞り込み
- バージョニング方針
- 認証方式
- レート制限
- 冪等性
- タイムアウト
- 監視可能なリクエストID

## データベースを選ぶ

データの関係とアクセス方法から選びます。

### リレーショナルデータベース

表、行、列でデータを管理し、SQLで操作します。

適した用途:

- 利用者、注文、商品など関係が明確
- トランザクションが必要
- 整合性を強く保ちたい
- 複数条件で検索・集計する

### ドキュメントデータベース

JSONに近い構造でデータを保持します。

適した用途:

- 項目構造が柔軟
- 一つのまとまりとして読み書きする
- 入れ子構造が多い

### キャッシュ・インメモリストア

頻繁に参照する値、セッション、短期間の状態などを高速に扱います。永続データの唯一の保存先として使えるかは製品設定によります。

データベース選定では、流行より次を確認します。

- 必要な整合性
- 検索パターン
- データ量と増加率
- バックアップ・復元
- 運用経験
- 移行方法
- 障害時の挙動
- 費用

## 認証と認可

認証は「誰か」を確認し、認可は「何をしてよいか」を判断します。

```text
認証:
この利用者はAliceである

認可:
Aliceは自分の記事を編集できるが、
他人の記事は削除できない
```

ログイン済みであることだけを確認し、対象データへの権限を確認しない実装は不十分です。すべての保護対象操作で、サーバー側の認可を行います。

セッションCookieを使う場合は、用途に応じて次を検討します。

- `Secure`
- `HttpOnly`
- `SameSite`
- 有効期限
- セッションIDの更新
- ログアウト時の無効化

独自の暗号方式やパスワード保存方式を設計せず、実績のある認証基盤とライブラリを使います。

## セキュリティ

OWASP Top 10は、Webアプリケーションで重大になりやすいリスクを理解する入口です。2026年6月時点ではOWASP Top 10:2025が公開されています。

### 入力を信用しない

ブラウザ側で制限しても、HTTPリクエストは直接作成できます。

- 型
- 長さ
- 形式
- 許可する値
- ファイルサイズ
- MIMEタイプ
- 利用者の権限

これらをサーバー側で検証します。

### SQLインジェクションを防ぐ

入力値をSQL文字列へ連結しません。

```python
# 避ける
query = f"SELECT * FROM users WHERE email = '{email}'"
```

プレースホルダーやORMの安全なAPIを使います。

```python
cursor.execute("SELECT * FROM users WHERE email = ?", (email,))
```

### XSSを防ぐ

利用者入力をHTMLとして出力しないことが基本です。テンプレートエンジンの自動エスケープを無効化せず、HTMLを許可する必要がある場合だけ、許可リスト方式で無害化します。

### CSRFを防ぐ

Cookieを認証に使う更新操作では、`SameSite`設定、CSRFトークン、Origin確認などを利用します。GETリクエストでデータを変更しません。

### 依存関係を管理する

- 不要な依存を追加しない
- ロックファイルを管理する
- 脆弱性情報を監視する
- 自動更新を検証して取り込む
- メンテナンスされていないパッケージを避ける
- パッケージのインストールスクリプトを無条件に信用しない

### セキュリティヘッダー

構成に応じて次を検討します。

- Content-Security-Policy
- Strict-Transport-Security
- X-Content-Type-Options
- Referrer-Policy
- Permissions-Policy
- frame-ancestors

ヘッダーは設定するだけでなく、本番レスポンスへ実際に付いているか確認します。

### 秘密情報

APIキー、パスワード、秘密鍵、接続文字列をGitへコミットしません。

- ローカル: Git管理外の環境変数ファイル
- CI/CD: Secrets管理機能
- 本番: ホスティング環境や秘密管理サービス

誤って公開した場合は、ファイルを削除するだけでは不十分です。漏えいした資格情報を失効・再発行します。

## アクセシビリティ

W3CはWCAG 2.2の利用を推奨しています。WCAGは次の4原則で構成されます。

| 原則     | 意味                           |
| -------- | ------------------------------ |
| 知覚可能 | 情報を利用者が認識できる       |
| 操作可能 | キーボードなどで操作できる     |
| 理解可能 | 内容と操作が理解できる         |
| 堅牢     | 支援技術を含む環境で解釈できる |

最低限、次を確認します。

- HTMLの`lang`を設定する
- 見出し構造を整える
- フォームへラベルとエラー説明を付ける
- 画像へ適切な代替テキストを付ける
- キーボードだけで操作できる
- フォーカス位置が見える
- 色だけで状態を伝えない
- 十分な色コントラストを確保する
- 動画へ字幕を付ける
- 自動再生や激しい動きを制御できる
- 拡大しても内容や操作を失わない
- エラー箇所と修正方法が分かる

自動検査だけでは、代替テキストの適切さや操作の分かりやすさを判断できません。キーボード操作、スクリーンリーダー、拡大表示などの手動確認も行います。

## Webパフォーマンス

Web Vitalsでは、利用者体験を測る主要指標としてCore Web Vitalsが定義されています。

| 指標 | 対象                     | 良好の目安    |
| ---- | ------------------------ | ------------- |
| LCP  | 主要コンテンツの表示速度 | 2.5秒以内     |
| INP  | 操作への応答性           | 200ミリ秒以下 |
| CLS  | 予期しないレイアウト移動 | 0.1以下       |

判定では、モバイルとデスクトップを分け、実利用データの75パーセンタイルを見ることが推奨されています。

### 改善の基本

- 配信するJavaScriptを減らす
- 不要なCSSを減らす
- 画像を適切なサイズと形式にする
- 画面外の画像を遅延読み込みする
- 画像の表示サイズを事前確保する
- フォントの種類とウェイトを絞る
- 圧縮を有効にする
- キャッシュを設定する
- CDNを利用する
- サーバー応答を短縮する
- 大きな処理を分割する
- サードパーティースクリプトを制限する

JavaScriptバンドルの最適化だけに注目せず、画像、フォント、サーバー処理、外部タグを含めて計測します。

### 計測

開発環境の高速なPCだけでは実態を把握できません。

- Lighthouseなどのラボ計測
- PageSpeed Insights
- Chrome DevTools
- 実利用者のRUM
- サーバーの応答時間
- APIの遅延分布

公開前のラボ計測と、公開後のフィールド計測を組み合わせます。

## SEO

検索エンジン最適化は、検索エンジンだけのためにページを作ることではありません。内容と構造を明確にし、利用者と検索エンジンが理解・発見しやすくします。

### 基本項目

- ページごとに固有の`title`
- 内容を表すmeta description
- 論理的な見出し構造
- 説明的なURL
- 内部リンク
- モバイル対応
- 高速な表示
- HTTPS
- canonical URL
- sitemap
- robots.txt
- 構造化データ
- 適切なHTTPステータス

### コンテンツ確認

検索語を不自然に繰り返すより、利用者の疑問へ正確に答えます。

- 誰向けの内容か
- 結論は何か
- 根拠は何か
- 情報は最新か
- 著者や運営者は明確か
- 関連ページへ移動できるか

検索エンジンへ表示した内容と利用者へ表示する内容を意図的に変える手法は避けます。

## プライバシー

アクセス解析、問い合わせ、会員登録、広告などで個人情報を扱う場合は、収集前に整理します。

- 何を収集するか
- なぜ必要か
- どこへ送信するか
- いつ削除するか
- 誰がアクセスできるか
- 第三者提供があるか
- Cookieを利用するか
- 利用者が拒否・削除を要求できるか

「後で使うかもしれない」という理由だけで情報を集めません。収集量を最小化し、保存期間を決めます。

法律上の要件は地域、事業、データ種類によって異なります。必要に応じて法務・専門家へ確認します。

## テスト

テストはコード量ではなく、失敗したときの影響に応じて設計します。

### 静的検査

- フォーマッター
- Linter
- 型検査
- HTML検証
- Markdown検証
- リンク検査
- 依存関係監査

### 単体テスト

関数や小さなモジュールを検証します。

```javascript
import { describe, expect, it } from "vitest";
import { calculateTotal } from "./price";

describe("calculateTotal", () => {
  it("数量と単価から合計金額を計算する", () => {
    expect(calculateTotal(3, 1200)).toBe(3600);
  });
});
```

### 統合テスト

API、データベース、外部サービスとの連携を検証します。

- 正常な登録
- 入力不正
- 権限不足
- 重複登録
- 外部API失敗
- トランザクションのロールバック

### E2Eテスト

実際のブラウザで、利用者の主要操作を検証します。

```text
トップページを開く
→ 商品を検索する
→ カートへ追加する
→ 注文内容を確認する
```

すべての細かな表示をE2Eテストへ入れると遅く不安定になります。売上、登録、ログインなどの重要経路へ絞ります。

### 手動確認

- PCとスマートフォン
- キーボード操作
- ライト・ダークテーマ
- 長い文章や大きな数値
- 空データ
- エラーメッセージ
- 低速通信
- 対応対象ブラウザ

## Gitと開発フロー

ソースコードはGitで管理します。

```text
main
  └─ feature/contact-form
       ├─ 実装
       ├─ テスト
       └─ Pull Request
```

基本的な流れは次のとおりです。

1. Issueやタスクで目的を定義する
2. 小さなブランチで変更する
3. 自動検査を実行する
4. Pull Requestでレビューする
5. CI成功後にマージする
6. 自動デプロイする
7. 本番を確認する

コミットには秘密情報、生成物、個人の環境設定を含めません。`.gitignore`を設定し、レビュー可能な大きさに変更を分けます。

## CI/CD

CIは、pushやPull Requestのたびに自動検査を行います。

```text
コード変更
  ↓
依存関係を復元
  ↓
Lint・型検査
  ↓
テスト
  ↓
ビルド
  ↓
セキュリティ・リンク検査
```

CDは、検証済みの成果物を環境へ配布します。

```text
mainへマージ
  ↓
本番ビルド
  ↓
成果物を保存
  ↓
本番へデプロイ
  ↓
ヘルスチェック
  ↓
必要ならロールバック
```

同じソースから同じ成果物を再現できるよう、依存バージョンをロックし、手動で本番サーバーのファイルを書き換えない運用にします。

## 公開先を選ぶ

必要な処理によって公開先が変わります。

| 構成                 | 必要な機能                               |
| -------------------- | ---------------------------------------- |
| 静的サイト           | 静的ファイル配信、CDN、独自ドメイン      |
| SSR                  | アプリケーション実行環境、スケーリング   |
| API                  | サーバーレス関数、コンテナ、常駐サーバー |
| データベース         | バックアップ、暗号化、接続制御           |
| ファイルアップロード | オブジェクトストレージ                   |
| メール送信           | メール配信サービス                       |

比較項目:

- 対応する実行環境
- リージョン
- 自動スケール
- コールドスタート
- 帯域と転送量
- ログ保持
- バックアップ
- 独自ドメインとTLS
- 障害時のサポート
- ベンダーロックイン
- 月額費用と従量課金

無料枠だけで選ぶと、利用増加後の制限や移行コストが問題になることがあります。本番運用に必要な条件で比較します。

## ドメインとHTTPS

独自ドメインを使う場合は、ドメイン管理サービスでDNSを設定します。

代表的なレコード:

| レコード | 用途                     |
| -------- | ------------------------ |
| A        | IPv4アドレスへ対応付ける |
| AAAA     | IPv6アドレスへ対応付ける |
| CNAME    | 別のホスト名へ対応付ける |
| MX       | メール配送先を指定する   |
| TXT      | 所有確認やメール認証など |

DNS変更は利用者へ届くまで時間がかかることがあります。TTL、旧設定、証明書発行の順序を考慮します。

インターネット公開サイトではHTTPSを前提とします。HTTPからHTTPSへリダイレクトし、混在コンテンツを残さないようにします。

## 監視と運用

デプロイ成功は運用開始を意味します。公開後に問題を検知できる仕組みが必要です。

### 監視対象

- HTTP応答と稼働率
- レスポンスタイム
- エラー率
- API失敗
- データベース接続
- CPU・メモリ
- キューの滞留
- 証明書期限
- ドメイン期限
- バックアップ結果
- Core Web Vitals

### ログ

ログには、調査に必要な情報を構造化して記録します。

```json
{
  "timestamp": "2026-06-15T10:00:00Z",
  "level": "error",
  "requestId": "req_123",
  "route": "/api/orders",
  "status": 500,
  "errorCode": "ORDER_SAVE_FAILED"
}
```

パスワード、APIキー、完全なクレジットカード番号、不要な個人情報をログへ記録しません。

### バックアップ

「バックアップを取っている」だけでは不十分です。

- 対象データ
- 取得頻度
- 保存期間
- 保存先
- 暗号化
- 復旧手順
- 復旧に必要な時間
- 復元テスト

実際に復元できることを定期的に確認します。

### 更新

- 依存パッケージ
- ランタイム
- OS
- コンテナイメージ
- TLS設定
- 外部API
- ブラウザ仕様
- 法令・ポリシー

更新を放置せず、自動通知、定期メンテナンス、検証環境を用意します。

## 段階的な学習順序

Web開発を初めて学ぶ場合は、次の順序が現実的です。

1. HTMLで文章、リンク、画像、フォームを作る
2. CSSのボックスモデル、Flexbox、Gridを学ぶ
3. レスポンシブデザインを実装する
4. JavaScriptの変数、関数、配列、DOM、イベントを学ぶ
5. `fetch()`とHTTP、JSONを学ぶ
6. GitとGitHubで変更を管理する
7. 静的サイトを公開する
8. アクセシビリティ、性能、SEOを計測する
9. 必要になってからフレームワークを学ぶ
10. バックエンド、API、データベースを学ぶ
11. 認証、認可、セキュリティを学ぶ
12. テスト、CI/CD、監視を導入する

最初からフロントエンドフレームワーク、バックエンド、データベース、コンテナ、クラウドを同時に学ぶと、問題の原因を切り分けにくくなります。小さな静的サイトを公開するところから始めます。

## 構築手順の実践例

### Phase 1: 目的と構造

1. 利用者と目的を一文で定義する
2. 必要なページを列挙する
3. ページ間の移動を図にする
4. 各ページの見出しと内容を書く
5. Must機能を決める

### Phase 2: HTML

1. ページ構造をHTMLで作る
2. 見出し、リンク、画像、フォームを配置する
3. キーボードで移動できるか確認する
4. JavaScriptなしで内容を確認する

### Phase 3: CSS

1. 基本文字と色を決める
2. コンテンツ幅と余白を決める
3. Flexbox・Gridでレイアウトする
4. スマートフォン幅を確認する
5. フォーカス表示とコントラストを確認する

### Phase 4: JavaScript

1. 必要な操作だけ追加する
2. 読み込み・成功・失敗状態を作る
3. 二重送信を防ぐ
4. エラーを利用者へ説明する
5. JavaScript失敗時の影響を確認する

### Phase 5: 品質

1. Lintとフォーマッターを導入する
2. 重要ロジックへ単体テストを追加する
3. 主要操作をブラウザで確認する
4. リンクと画像を検査する
5. Lighthouseなどで計測する
6. セキュリティレビューを行う

### Phase 6: 公開

1. 本番用ビルドを作る
2. CI/CDを設定する
3. ドメインとHTTPSを設定する
4. robots.txtとsitemapを確認する
5. 404ページを確認する
6. 公開URLで主要操作を確認する

### Phase 7: 運用

1. 監視とアラートを設定する
2. バックアップと復元を確認する
3. 更新担当者と手順を決める
4. 利用状況とエラーを確認する
5. 定期的に依存関係を更新する

## よくある失敗

### 要件より先にフレームワークを決める

採用したい技術に合わせて要件を変えると、不要な複雑さが増えます。必要なデータ更新、対話性、運用条件を先に決めます。

### デザインを画像のように固定する

文章量、翻訳、文字拡大、画面幅の変化で崩れます。コンテンツの変化を許容するレイアウトにします。

### スマートフォン確認を最後にする

後からモバイル対応すると、HTML構造やナビゲーションから作り直す場合があります。小さい画面を初期段階から確認します。

### セキュリティを公開直前に確認する

認証、データ構造、権限設計は後付けが難しい領域です。設計段階から脅威と権限境界を考えます。

### アクセシビリティをARIAだけで解決する

標準HTML要素を正しく使うことが基本です。ARIAは不足する意味を補うものであり、不適切なHTML構造を自動修正するものではありません。

### 自動テストだけで品質を判断する

自動テストが通っても、文章が理解できるか、操作が迷わないか、視覚的に崩れていないかは別問題です。手動確認と利用者テストも必要です。

### 本番環境を手作業で変更する

変更履歴と再現性が失われます。コードと設定をGitで管理し、CI/CDから配布します。

### 公開後の担当者を決めない

情報、依存関係、証明書、ドメインは更新が必要です。所有者、更新頻度、障害時の連絡先を決めます。

## 公開前チェックリスト

### 内容

- サイトの目的と対象利用者が明確
- タイトル、説明、連絡先が正しい
- 誤字と古い情報を確認した
- 404ページがある
- 利用規約やプライバシーポリシーが必要か確認した

### HTML・操作

- `lang`が正しい
- 見出し構造が論理的
- 画像に適切な`alt`がある
- フォームにラベルがある
- キーボードだけで操作できる
- フォーカス位置が見える
- エラー内容と修正方法が分かる

### レスポンシブ

- スマートフォンで横にはみ出さない
- 表とコードを閲覧できる
- タップ対象が小さすぎない
- 文字を拡大しても操作できる
- 長い文字列やURLで崩れない

### セキュリティ確認

- HTTPSを使用する
- 秘密情報を成果物へ含めていない
- サーバー側で入力検証する
- 認証と認可を分けて確認した
- SQLを文字列連結していない
- 利用者入力を安全に出力する
- Cookie属性を確認した
- 依存関係を監査した
- セキュリティヘッダーを確認した

### 性能

- 画像を適切に圧縮した
- 不要なJavaScriptを削減した
- キャッシュを設定した
- LCP、INP、CLSを計測した
- 低速通信と低性能端末を考慮した

### SEO確認

- ページごとに固有のタイトルと説明がある
- canonical URLが正しい
- sitemapとrobots.txtが正しい
- 内部リンクが切れていない
- 構造化データが必要か確認した
- 適切なステータスコードを返す

### 公開・運用

- CIが成功する
- 本番ビルドを再現できる
- 独自ドメインとDNSが正しい
- 公開URLを直接再読み込みできる
- 監視とアラートがある
- バックアップと復元手順がある
- ロールバック方法がある
- 更新担当者が決まっている

## まとめ

Web構築の中心は、特定のフレームワークを覚えることではありません。

1. 利用者と目的を定義する
2. 必要な構成だけを選ぶ
3. HTMLで意味のある構造を作る
4. CSSで柔軟なレイアウトを作る
5. JavaScriptを必要な機能へ限定する
6. サーバー側でデータと権限を守る
7. アクセシビリティ、性能、SEOを設計へ含める
8. テストとCI/CDで変更を安全に公開する
9. 監視、更新、バックアップを継続する

小さく構築して実際に公開し、利用状況と失敗から改善する方法が現実的です。要件を満たす最も単純な構成から始め、必要性が確認できた段階でフレームワーク、バックエンド、データベースなどを追加します。
